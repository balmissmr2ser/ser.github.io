SERVICIOS DE CORREO EN LINUX

INTRODUCCIÓN

Un sistema de correo electrónico no es un único servicio, sino un conjunto de procesos que colaboran para aceptar mensajes, encaminarlos, almacenarlos y ponerlos a disposición del usuario final. En un despliegue clásico de Linux, lo habitual es separar al menos dos piezas:

  • Un servicio SMTP para el transporte y entrega entre servidores o clientes.
  • Un servicio IMAP/POP3 para que el usuario pueda consultar su buzón.

En esta unidad se trabajará con Postfix como servidor SMTP y con Dovecot como servidor de acceso a buzones. Es una combinación habitual en entornos Debian y Ubuntu porque ofrece una configuración razonablemente clara, buen rendimiento y una superficie de ataque moderada si se endurece de forma correcta.

ARQUITECTURA DEL SERVICIO DE CORREO

COMPONENTES PRINCIPALES

Componente Función Ejemplo en Linux
MTA Transfiere mensajes entre servidores y acepta correo entrante o saliente. Postfix
MDA Deposita el mensaje en el buzón del usuario. Dovecot LDA o entrega local de Postfix
MUA Cliente de correo usado por la persona usuaria. Thunderbird, Outlook, mutt
DNS Publica los registros necesarios para localizar el servidor de correo. Registros MX, A y PTR

PROTOCOLOS Y PUERTOS HABITUALES

Servicio Protocolo Puerto TCP Uso habitual
SMTP SMTP 25 Transporte entre servidores y recepción en el MTA
Submission SMTP autenticado 587 Envío de correo por clientes autenticados
SMTPS SMTP sobre TLS implícito 465 Compatibilidad con algunos clientes
POP3 POP3 110 Descarga de mensajes al cliente
POP3S POP3 sobre TLS 995 POP3 cifrado
IMAP IMAP 143 Acceso remoto sincronizado al buzón
IMAPS IMAP sobre TLS implícito 993 IMAP cifrado

En la práctica moderna, el acceso de clientes suele hacerse mediante submission 587 para el envío y IMAP 143/993 o POP3 110/995 para la lectura. El puerto 25 se reserva principalmente para comunicaciones entre MTAs.

REQUISITOS PREVIOS

IDENTIDAD DEL SERVIDOR

Antes de instalar los servicios conviene fijar correctamente el nombre del sistema. Un servidor de correo debe tener un nombre coherente y resoluble, por ejemplo mail.ser.smr.

hostnamectl set-hostname mail.ser.smr

Además, /etc/hosts debe incluir al menos una resolución local coherente para el propio servidor:

127.0.0.1       localhost
192.168.1.201   mail.ser.smr mail

DNS NECESARIO

Para que otros sistemas localicen el servidor de correo del dominio hacen falta, como mínimo, estos registros:

ser.smr.        IN MX 10 mail.ser.smr.
mail.ser.smr.   IN A  192.168.1.201
201.1.168.192.in-addr.arpa. IN PTR mail.ser.smr.

Sin un registro MX correcto, los MTAs remotos no sabrán a qué servidor deben entregar el correo del dominio. Sin resolución A y PTR coherente es frecuente que aparezcan rechazos o puntuaciones negativas por filtros antispam.

INSTALACIÓN DE POSTFIX Y DOVECOT

PAQUETES BÁSICOS

apt update
apt install postfix dovecot-core dovecot-imapd dovecot-pop3d mailutils

Durante la instalación de Postfix en Debian suele aparecer un asistente. Para un laboratorio sencillo se puede elegir Internet Site e indicar como nombre del sistema mail.ser.smr.

COMPROBACIÓN DE SERVICIOS

systemctl status postfix
systemctl status dovecot
ss -ltnp | egrep ':(25|110|143|587|993|995) '

CONFIGURACIÓN BÁSICA EN MAIN.CF

Una configuración inicial razonable para un servidor del dominio ser.smr con buzones locales en formato Maildir puede ser la siguiente:

myhostname = mail.ser.smr
mydomain = ser.smr
myorigin = $mydomain
inet_interfaces = all
inet_protocols = ipv4
mydestination = $myhostname, localhost.$mydomain, localhost, $mydomain
mynetworks = 127.0.0.0/8, 192.168.1.0/24
home_mailbox = Maildir/
smtpd_banner = $myhostname ESMTP
append_dot_mydomain = no
recipient_delimiter = +
relayhost =
smtpd_relay_restrictions = permit_mynetworks, permit_sasl_authenticated, reject_unauth_destination
alias_maps = hash:/etc/aliases
alias_database = hash:/etc/aliases

Los puntos más importantes de esta configuración son los siguientes:

  • myhostname identifica al servidor SMTP.
  • mydestination define para qué dominios el servidor es destino final, es decir, dónde reparte el correo localmente.
  • mynetworks indica qué redes pueden usar el servidor como relay sin autenticación. Debe ser muy restrictivo.
  • home_mailbox = Maildir/ hace que cada usuario guarde sus mensajes en ~/Maildir.
  • reject_unauth_destination evita convertir el servidor en un open relay.

DOVECOT

OBJETIVO DE DOVECOT

Dovecot se encarga de que los clientes puedan leer y gestionar sus buzones mediante IMAP o POP3. También puede participar en la autenticación SMTP y en la entrega local, aunque para un escenario básico es suficiente con usarlo como servidor de acceso a buzones.

CONFIGURACIÓN MÍNIMA

En Debian y Ubuntu la configuración se reparte entre varios ficheros dentro de /etc/dovecot/conf.d/. Los cambios básicos suelen concentrarse en 10-mail.conf, 10-auth.conf y 10-master.conf.

MAILDIR

# /etc/dovecot/conf.d/10-mail.conf
mail_location = maildir:~/Maildir

PROTOCOLOS HABILITADOS

# /etc/dovecot/dovecot.conf
protocols = imap pop3

AUTENTICACIÓN CON USUARIOS DEL SISTEMA

# /etc/dovecot/conf.d/10-auth.conf
disable_plaintext_auth = yes
auth_mechanisms = plain login

La directiva disable_plaintext_auth = yes obliga a proteger la autenticación con TLS cuando se usan mecanismos simples como PLAIN o LOGIN.

INTEGRACIÓN DE DOVECOT CON POSTFIX PARA AUTH SMTP

Si se quiere permitir el envío autenticado por el puerto 587, Postfix puede delegar la autenticación SASL en Dovecot. Para ello hay que crear un socket accesible desde el entorno chroot de Postfix.

# /etc/dovecot/conf.d/10-master.conf
service auth {
  unix_listener /var/spool/postfix/private/auth {
    mode = 0660
    user = postfix
    group = postfix
  }
}
# /etc/postfix/main.cf
smtpd_sasl_type = dovecot
smtpd_sasl_path = private/auth
smtpd_sasl_auth_enable = yes
broken_sasl_auth_clients = yes

SERVICIO SUBMISSION EN MASTER.CF

submission inet n       -       y       -       -       smtpd
  -o syslog_name=postfix/submission
  -o smtpd_tls_security_level=encrypt
  -o smtpd_sasl_auth_enable=yes
  -o smtpd_recipient_restrictions=permit_sasl_authenticated,reject

De esta forma, los clientes no usarán el puerto 25 para enviar correo saliente, sino el 587 con autenticación y cifrado.

ESCENARIO PRÁCTICO: SERVIDOR CON USUARIOS LOCALES

CREACIÓN DE USUARIOS

adduser admin
adduser alumno1
adduser alumno2

CREACIÓN DE BUZONES MAILDIR

Si el buzón aún no existe, se puede crear el árbol Maildir desde cada cuenta o con la utilidad de Dovecot.

doveadm mailbox create -u admin INBOX
doveadm mailbox create -u alumno1 INBOX
doveadm mailbox create -u alumno2 INBOX

Otra posibilidad es generar ~/Maildir enviando el primer mensaje local una vez configurado home_mailbox = Maildir/.

ENVÍO DE PRUEBA LOCAL

echo "Prueba de correo local" | mail -s "mensaje 01" alumno1

REVISIÓN DE REGISTROS

tail -f /var/log/mail.log

SEGURIDAD Y ENDURECIMIENTO

  • No permitir relay abierto. La directiva crítica es reject_unauth_destination.
  • Publicar submission 587 para clientes y reservar el 25 para tráfico entre servidores.
  • Usar TLS en SMTP, IMAP y POP3. En prácticas se puede empezar con certificados locales; en producción conviene usar una CA pública o corporativa.
  • Limitar con firewall los puertos realmente necesarios.
  • Revisar de forma periódica /var/log/mail.log para detectar errores de autenticación, colas bloqueadas o rechazos por DNS.
  • Asegurar coherencia entre hostname, registros A, MX y PTR.
  • Evitar cuentas sin contraseña y deshabilitar accesos innecesarios.

Con esta base ya se dispone de un servidor de correo funcional para un entorno de laboratorio: Postfix recibe y entrega mensajes del dominio, Dovecot ofrece acceso a los buzones y el sistema puede validarse mediante pruebas SMTP, IMAP, POP3 y consultas DNS. A partir de aquí se puede evolucionar hacia escenarios más avanzados con dominios virtuales, antispam, antivirus, webmail o integración con LDAP.
